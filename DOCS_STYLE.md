# 中文优先文档规范

## 适用范围

1. 这份规范适用于仓库内所有 Markdown 文档，尤其是 `src/content/docs/**/*.md`。
2. 目标是让后续文档任务可以直接按规则写作，不需要额外找人确认写法。
3. 文档以中文为主，保留常见英文技术术语原文。

## 写作总则

1. 正文优先使用中文叙述，术语首次出现时可以写成 `中文（English term）`。
2. 代码标识符、命令名、文件名、错误消息、协议名、包名，保持原文，不要硬翻译。
3. 常见技术词保持业界通用写法，例如 `HTTP`、`HTTPS`、`TLS`、`SSH`、`Kubernetes`、`systemd`、`CLI`、`trait`、`RISC-V`。
4. 不要写空话，也不要把不确定的内容写得像确定事实。

### Frontmatter 要求

1. `src/content/docs/**/*.md` 页面应保留 frontmatter。
2. `title` 必填。
3. `description` 优先补齐，尤其是目录页和概览页。
4. 目录页按需要使用 `template: splash`，不要把它当成普通正文页的默认项。

### 标题层级

1. 每页只保留一个 `#` 级标题，通常就是页面标题。
2. 主要章节用 `##`，子主题用 `###`，更深层次再用 `####`。
3. 不要跳级写标题，比如直接从 `##` 跳到 `####`。
4. 目录页和主题切换处可以用 `---`，但不要为了装饰频繁插分割线。

### 相对链接与 Starlight 路由

1. 同目录或同集合内优先用相对链接，并且不要写 `.md` 后缀，例如 `[系统服务管理](2-service_manager)`。
2. 不要把源文件路径当成站点路由。
3. 跨目录链接前先确认目标路由确实存在，不能凭文件名猜路径。
4. 如果路由没核实清楚，先写成 `待核验`，不要先编一个看起来合理的链接。

## 引用与参考资料

1. 文内事实句尽量配来源，格式可以写成 `（来源：...）` 或 `（参考：...）`。
2. 一段话依赖多个来源时，用分号分开，别把来源藏到段末不说明。
3. 页面尾部可放一个独立的 `## 参考资料` 区块，只列本文实际用到的来源。
4. 来源优先级固定为 `official docs`，`man pages`，`upstream project docs`，最后才是 `reputable secondary references`，而且只有在前面三类不够用时才考虑。
5. 安全、版本、兼容性、默认值、弃用状态、权限边界这类说法，正文必须有权威来源。没有权威来源就写 `待核验`，不要猜。
6. 版本稳定且会变化的网页，引用时要保留访问日期。

参考资料页面格式示例：

```md
## 参考资料

1. [systemd.unit(5)](https://man.archlinux.org/man/systemd.unit.5)（访问日期：2026-05-31）
2. [Kubernetes Nodes and swap memory](https://kubernetes.io/docs/concepts/architecture/nodes/#swap-memory)（访问日期：2026-05-31）
3. [Neovim API](https://neovim.io/doc/user/api.html)（访问日期：2026-05-31）
```

## 版本与时间

1. 如果文档讲的是特定版本，就直接写清楚适用版本，例如 `适用版本：v1.30`、`Ubuntu 24.04`、`systemd 255`。
2. 如果引用的是当前稳定版官方文档，就写访问日期，避免把会变化的网页当成固定事实。
3. 版本信息来自文档本身时，正文以该版本为准，不要偷偷换成别的版本。
4. 如果版本信息和权威来源对不上，先标 `待核验`。

## 安全/命令说明

1. 只要命令可能删除、覆盖、格式化、重置、重启、停服、改防火墙、改证书、改密钥、提权、改集群状态，就要先讲后果，再给命令。
2. 涉及 `sudo`、root、`systemctl`、`journalctl`、`iptables`、`nft`、`firewalld`、`ssh`、`openssl`、`kubeadm`、`kubectl`、`containerd`、`runc`、`mkfs`、`fdisk`、`mount`、`rm -rf` 的命令，默认按高风险处理。
3. 涉及密码、token、cookie、私钥、证书、kubeconfig、known_hosts、authorized_keys 的例子，一律用占位符，不要放真实值。
4. 有只读或预览方式时，优先先写只读方式，再写会改状态的方式。
5. 任何 TLS、SSH、K8s、Linux 运维命令，都要写清楚前提条件和影响范围。
6. 不要把危险命令包装成“顺手执行”或“通常没事”。

## 文档拆分

1. 一页只讲一个主题。多个互不依赖的主题，拆成多个页面。
2. 长页、命令密集页、证书或权限类页面、协议细节页，优先考虑分拆成总览页加子页。
3. 拆分时先保留目录和跳转，不要直接删掉有用内容。
4. 旧内容如果还可能有参考价值，就保留、修正或标记，不要默认删除。
5. 只有确认内容重复、无效、且没有保留价值时，才考虑清理。

## 废弃/待核验标记

1. `待核验` 用在事实不够确定、来源不够权威、或者版本不清楚的地方。
2. `已过时` 用在内容还保留，但已经不是当前推荐做法的地方。
3. `已弃用` 用在官方已经不再推荐或明确废弃的 API、命令、配置项。
4. 不要把 `待核验` 擅自改成确定句式。
5. 发现内容过时后，优先补正确版本、替代方案和参考来源，再决定是否降级为历史说明。

## 页面末尾模板

```md
## 参考资料

1. [来源 1](https://example.com)（访问日期：2026-05-31）
2. [来源 2](https://example.org)（访问日期：2026-05-31）
```
